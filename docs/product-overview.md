# 产品方案

## 1. 产品名称

正式名称：**ReviewLume**

VS Code Marketplace 展示名建议：

> ReviewLume – AI Code Review Pack

不建议把 ChatGPT、OpenAI、GPT、Sol、Claude 或 Gemini 放入产品正式名称。兼容服务可在功能描述中准确说明，并附第三方独立项目免责声明。

## 2. 目标用户

- 使用多个 AI 模型协作开发的个人开发者。
- 使用一个模型实施、另一个模型最终审核的团队。
- 不愿把完整代码库直接交给 AI 的开发者。
- 希望保留审核证据和修复闭环的项目维护者。

## 3. 核心痛点

### 3.1 上下文失控

直接让 AI 读取整个项目会造成大量无关上下文、成本增加和审核目标偏移。

### 3.2 敏感信息风险

项目中可能包含 `.env`、证书、生产配置、数据库、访问令牌和客户数据。

### 3.3 审核结果不可追踪

AI 的回答往往停留在聊天窗口，缺少问题级别、证据、处理状态和二次复核记录。

### 3.4 网页 AI 与本地工程割裂

网页版模型能力强，但无法安全、稳定地理解本地改动范围。

## 4. 产品原则

- **Focused**：只收集本次审核真正需要的上下文。
- **Private**：发送前扫描并阻止敏感内容。
- **Reviewable**：所有要发送的内容都可预览。
- **Provider-neutral**：不绑定单一 AI 服务。
- **Human-in-the-loop**：关键动作由用户确认。
- **Auditable**：保留审核请求、回复、结论和处理记录。

## 5. 用户故事

### US-01 生成审核包

作为开发者，我希望选择一个 Git 提交范围，自动获得 diff、变更文件、关联测试和验收要求，以便交给 AI 审核。

### US-02 敏感扫描

作为开发者，我希望发送前看到敏感文件和疑似密钥警告，避免泄露凭据。

### US-03 自定义审核模板

作为开发者，我希望选择快速、标准、高风险或自定义模板，以匹配不同任务风险。

### US-04 导入审核结论

作为开发者，我希望把 AI 回答整理成阻塞问题、非阻塞问题和测试缺口，并保存到项目外的本地历史中。

### US-05 二次复核

作为开发者，我希望根据首次审核问题和修复报告自动生成复核提示，防止问题遗漏。

## 6. 功能分层

### 核心层

- Git 范围选择
- 文件收集
- 排除规则
- 敏感扫描
- Review Pack 生成
- 提示模板
- 报告导入
- 审核历史

### 可选桥接层

- 打开 AI 网页
- 自动填入提示
- 用户点击发送
- 用户主动导入当前回答

### 不进入近期规划

- 通用编码 Agent
- 自动写代码
- 自动执行命令
- 自动登录网页
- 网页内部接口适配
- 多账号管理
- 云端代码上传
